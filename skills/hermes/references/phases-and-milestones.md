# HERMES Phases, States, and Milestones

A reference to the HERMES phase model (traditional and agile), milestones as quality gates, and the full catalogs of milestones and states.

## Phase Model Overview

The phase model builds on the project life cycle. Every project starts with the Initiation phase and ends with the Closure phase. These two phases are common to both approaches and bracket the solution development phases in between. A project runs in either five phases (traditional approach) or three phases (agile approach). The approach itself is chosen during Initiation; within a single program some projects may run traditionally and others agilely.

- Traditional approach: Initiation, Concept, Implementation, Deployment, Closure.
- Agile approach: Initiation, Execution, Closure.

Phases start and end with milestones that act as quality gates (see "Milestones as Quality Gates" below).

## Traditional Phases

### Initiation
- Goal: provide a structured orientation for a focused project. It formulates which possible solutions exist and which path to take.
- Key outcomes: the suitable solution option is selected, along with a suitable scenario for the development procedure; the choice of traditional vs agile approach is decided here; the specific procedure and solution development are planned.
- Exit decision: phase release to begin the next phase.

### Concept
- Goal: define the concept(s) needed to develop the solution (for example product concept, solution architecture, ISDP concept), forming the prerequisite for development or adaptation.
- Key outcomes: concepts and the solution architecture that serve as the basis for building or adapting products, systems, and services.
- Exit decision: phase release to proceed to Implementation.

### Implementation
- Goal: develop or adapt the systems, products, services, organization, and interfaces according to the agreed concepts and architecture.
- Key outcomes: the system / product developed or adapted, interfaces realized, organization implemented, migration procedure realized, ready for testing and preliminary acceptance.
- Exit decision: phase release (with preliminary acceptance as a related decision) to proceed to Deployment.

### Deployment
- Goal: carry out the deployment measures, migration, and activation so the solution can be put into productive operation.
- Key outcomes: deployment measures carried out, migration carried out, operation activated, solution activated for users.
- Exit decision: closure phase release, which completes the provision of goods or services within solution development and allows the closure work to begin.

### Closure
- Goal: end the project and define the organizational and administrative transition from the project organization to the application organization.
- Key outcomes: project formally concluded; transition handled.
- Exit decision: project closure.

## Agile Phases

The agile approach uses the same Initiation and Closure phases as the traditional approach, with a single solution development phase in between.

### Initiation
- Goal and outcomes: identical role to the traditional Initiation phase (structured orientation, solution option and scenario selected, approach chosen, procedure planned).
- Exit decision: phase release.

### Execution
- Goal: develop the solution iteratively in releases. This single phase replaces the traditional Concept, Implementation, and Deployment phases.
- Key outcomes: working releases of the solution; optional interim releases may be approved, each creating an additional milestone.
- Exit decision: execution release governs the start of work per the execution order; release decisions govern each release; closure phase release leads into Closure.

### Closure
- Goal and outcomes: identical to the traditional Closure phase (end the project, transition from project organization to application organization).
- Exit decision: project closure.

## Milestones as Quality Gates

The source states that phases start and end with milestones, and that these milestones correspond to quality gates at which the outcomes and the procedure are decided. Deciding at a quality gate also involves coordination with the strategic objectives and requirements of the core organization. Under the agile approach, optional interim releases (analogous to phase releases) may be approved, creating additional milestones.

Key principles:
- All milestones are outcomes that mark decision points over the course of the project.
- Each decision task ends with a milestone.
- Depending on the module, there are different decisions and consequently different further milestones.
- A milestone is itself a state and is always the consequence of a decision; it marks and defines a specific point in time reached in the project, serving as an envisaged and achieved decision outcome for project steering and management.

## Milestone Catalog (4.4.4)

Each entry below names the milestone and what its decision enables.

- Acceptance milestone: decision on acceptance is made; the solution is definitively transferred to the application organization (and, where applicable, the operating organization). Warranty and regular operation begin.
- Migration acceptance milestone: decision on acceptance of migration is made; use of the new system can be released for users (decide on launch of operation).
- Tender milestone: decision on call for tenders is made; the tender can be published.
- Launch of operation milestone: decision on launch of operation is made; the solution can be put into operation.
- Execution release milestone: decision on execution release is made; work starts according to the execution order.
- ISDP concept milestone: decision on ISDP concept is made; implementation of the ISDP measures can begin.
- Solution architecture milestone: decision on solution architecture is made; this forms the prerequisite for the development or adaptation of systems.
- Phase release milestone: decision on phase release is made; work in the next phase can begin.
- Closure phase release milestone: decision on closure phase release is made; provision of goods or services within solution development is completed and work in the closure phase can begin.
- Product concept milestone: decision on product concept is made; the concept is the prerequisite for development or adaptation of products or services.
- Project closure milestone: decision on project closure is made; the project is concluded.
- Project initiation release milestone: decision on project initiation release is made; the project is formally launched and work starts according to the project initiation order.
- Release milestone: decision on release is made; work on the next release can begin under the agile approach.
- Preliminary acceptance milestone: decision on preliminary acceptance is made; this forms the basis for executing deployment measures and launch of operation.
- Next steps milestone: decision on next steps is made; this forms the basis for drawing up the project management plan and the execution order.
- Contract award milestone: decision on contract award is made; the decision can be published and contract work with the successful tenderer can begin.

## States Catalog (4.4.3)

States are intermediate outcomes that mark progress over the course of the project. (A milestone is a special kind of state that always results from a decision.) Each entry below names the state and its one-line meaning.

- Legacy system removed: the old system version and legacy system are decommissioned or removed per specification, including destruction or archiving of data.
- Operation activated: operation of the activated system is launched and run by the operating organization defined in the operating concept, with SLA measurement prerequisites met.
- Operating infrastructure realized: all infrastructure required to create and operate the system across environments (development, test, production) plus supervision components (monitoring, alarms, statistics) is in place.
- Operating organization realized: the operating organizational structure and processes are realized and the operating staff are trained to perform operating tasks.
- Deployment measures carried out: the measures realized in the deployment concept are executed, reviewed, and quality-assured (for example user training carried out with course assessments available).
- Deployment measures realized: the measures and organization needed for deployment are realized (for example superusers recruited and trained, or training materials produced).
- ISDP measures realized: the ISDP measures are realized on the basis of the ISDP concept, ensuring protection-need requirements are met.
- ISDP concept transferred: the ISDP concept is updated, checked by controlling and compliance bodies, and transferred from the project organization to the core organization.
- Migration carried out: migration from the old to the new or further-developed system is carried out and documented per specification; successful migration is the prerequisite for migration acceptance and the launch-of-operation decision.
- Migration procedure realized: the migration procedure is realized and tested by the developer or the developer's tester, with test evidence provided; a prerequisite for preliminary acceptance.
- Organization activated: the new organization is activated and runs its processes per the process descriptions; under traditional development this is a prerequisite for activating the product or system (agile activates only the relevant part).
- Organization implemented: the organization defined in the organization concept is implemented (position assignments, recruitment, etc.) based on the process and organization descriptions.
- Product activated: the activated product is made available to users for utilization.
- Product developed or adapted: the developed or adapted product is tested by the developer or the developer's tester and handed over to the user for tests and preliminary acceptance.
- Prototype realized: a prototype is realized to check feasibility or behavior and to assess and reduce risks; several prototypes (disposable or reusable) may be realized over the project.
- Interfaces realized: interfaces ensure data exchange with peripheral systems; tested by the developer or integrator and the developer's tester, then handed to the operator for integration.
- System activated: the activated system is made available to users for utilization.
- System developed or parameterized: the developed or parameterized system is tested by the developer or integrator and the developer's tester, then handed to the operator for integration; may proceed in several steps or releases.
- System integrated: the integrated system is available to the user's testers for testing and preliminary acceptance; it is activated after the preliminary acceptance and launch-of-operation decisions.
- Test infrastructure realized: a test infrastructure (test system, test data, test tools) is provided per the test concept, based on the realized operating infrastructure and matching the production system closely enough for realistic test cases; ISDP requirements apply if non-anonymized productive data is used.
- Test infrastructure transferred: the entire test infrastructure, including the test concept, is transferred to the core organization.

## Gaps

- The source range gives the goal and exit decision of each phase only in narrative form; per-phase outcome lists above are reconstructed from the milestone and state catalogs and from the narrative, not from an explicit per-phase outcome table in this range.
- Section 4.4.2 (checklists) appears in the read range but describes review-point checklists, not milestones or states; it is not catalogued here beyond noting that each decision/milestone has a corresponding checklist.
- One milestone (4.4.4.5 Execution release) and one state (4.4.3.x) reference the agile execution order; the precise list of which milestones apply to the agile vs traditional approach is not enumerated in this range.
