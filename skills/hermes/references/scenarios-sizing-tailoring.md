# HERMES Scenarios, Sizing, and Tailoring Reference

This reference explains what a HERMES scenario is, the catalog of standard scenarios and the modules each contains, how sizing derives a project's size from its project value, and how tailoring further customizes a scenario in the initiation phase.

## What a Scenario Is

A scenario is geared towards the implementation of projects of a specific nature (for example the development or adaptation of an IT solution, or an organizational adjustment). A scenario contains precisely those HERMES method components (modules, with their tasks and outcomes) that are important for developing the solution for that project type, which is what makes HERMES quick and easy to use.

Key properties:

- A scenario maps the complete solution development of a project and supports the project manager in execution planning.
- A scenario describes only solution development. The initiation phase and the closure phase are outside the scenario and are not part of any scenario.
- A scenario is built from modules. A scenario can be formed from whole modules and from parts of modules. A module can be used in several scenarios and, because a module can be more comprehensive than a scenario, a module can be used only partially in a scenario.
- Because modules are composed of their associated tasks and outcomes, each scenario comes with a coordinated solution development template, including the necessary document templates.
- The findings of the study (produced in the initiation phase) are decisive for selecting the appropriate standard scenario.
- HERMES users can adapt standard scenarios to the needs of their organization and project and create their own user-defined scenarios. User-defined scenarios can be shared with other HERMES users or made available to all users.

Each scenario has a scenario description with a uniform structure:

- Applicability: concrete project criteria for which the scenario is suitable.
- Modules: enumerates and graphically represents all modules of the scenario along the solution development. The project foundations module and parts of other modules that are used outside solution development (and so do not appear in any scenario) are shown shaded white.

## Catalog of Standard Scenarios

HERMES offers five standard scenarios for projects with different characteristics. The range of standard scenarios is not exhaustive: demand for new scenarios is reviewed periodically and additional standard scenarios are made available as needed. The five standard scenarios are:

- Service/product development
- Service/product adaptation
- IT development
- IT adaptation
- Organizational adjustment

### Scenario directory (applicability and modules)

Service/product development

- Applicability: projects where a service or product is developed and provided. Examples: development of training materials and courses on a specific topic, development of an internal standard, establishment of a delivery service.
- Modules: Project steering, Project management, Organization, Product, Deployment organization.

Service/product adaptation

- Applicability: projects where a product or service available on the market is procured, adapted, and integrated into the organization.
- Modules: Project steering, Project management, Procurement, Organization, Product, Deployment organization.
- Note: as needed, the Tests module can also be embedded in this scenario to organize and conduct tests of solutions.

IT development

- Applicability: projects where a new IT application is developed for the specific needs of one or more specialist areas (user needs), or an existing IT solution is further developed and is integrated both technically and organizationally.
- Modules: Project steering, Project management, Organization, IT system, Tests, Deployment organization, IT migration, IT operation, ISDP.

IT adaptation

- Applicability: projects where an IT solution available on the market (for example standard software or IT infrastructure) is procured, adapted, and integrated both technically and organizationally.
- Modules: Project steering, Project management, Procurement, Organization, IT system, Tests, Deployment organization, IT migration, IT operation, ISDP.

Organizational adjustment

- Applicability: projects where new organizations are established or existing organizations are adjusted through restructuring and innovation, new business areas, in- and outsourcing, acquisitions, mergers and separations, liquidations, (international) expansions, etc. Examples: relocation, adjustment, or creation of an organization; merger of organizations; outsourcing of services to a service center.
- Modules: Project steering, Project management, Organization, Deployment organization.

## Mandatory Modules (Project Governance)

To comply with project governance, the following modules must always be included in each project:

- Project steering
- Project management
- Project foundations
- Deployment organization

Project foundations occurs only in the initiation phase, so it does not appear in the scenario module table (scenarios cover only solution development). Customized modules can supplement the standard modules: organizations can develop their own subject-, organization-, or project-specific modules with existing or new customized tasks and outcomes and integrate them into their own projects or scenarios, supported by HERMES online.

## Adjusting a Scenario

A selected scenario is adjusted to the project in two steps, both supported by HERMES online:

1. Sizing: adapts the scenario to the determined size of the project.
2. Tailoring: optionally customizes the (sized) scenario further to the specific project requirements.

The general flow: the standard scenario is first adjusted via the sizing function according to the actual project value determined, producing a user-defined scenario fitted to the project size. In a second step, that scenario, once adapted to the appropriate scope, can optionally be further customized using the tailoring function. At the end, the customized scenario can be downloaded, including the adapted structure plans and templates.

## Sizing

Purpose: the full scope of HERMES project management is designed for large projects with a high level of complexity, to ensure completeness of both the information and the method itself. Most core organizations predominantly manage medium and small projects, so the application of the method should be adapted to the size of the project using the sizing function. Sizing reduces the complexity of the procedure and the scope of the documentation to an absolute minimum while still complying with HERMES governance.

How sizing works:

- The sizing function determines the actual required project size from several parameters that can be entered by the user. The aggregate result is referred to as the project value.
- Based on the determined project value, the sizing function provides the project manager with a scenario tailored to the size of the project, together with an execution structure plan and the necessary templates.
- This online-tool-supported adaptation guarantees method continuity and coherence (HERMES governance) and allows all projects to be processed in a lean manner.
- The resulting user-defined scenario can then optionally be adjusted to the respective project requirements using the tailoring function.

When sizing happens: in the initiation phase. A suitable scenario is selected and the project value is determined as needed as part of the decision on next steps (which clarifies solution option, scenario, approach traditional or agile, and project value). The decision on next steps is the prerequisite for drawing up the project management plan and the execution order.

## Tailoring

Purpose: using tailoring, standard scenarios or user-defined scenarios already created through sizing are adapted to the needs of the project and further customized. This can be done with the help of HERMES online.

There are four basic tailoring options, which can be used in combination:

1. Remove modules from an existing scenario: unnecessary modules are removed. Example: in a project without migration, remove the IT migration module from the IT development scenario.
2. Remove tasks and outcomes: the content of a module can be reduced by removing outcomes and their associated tasks, with the exception of the minimum required documents (these cannot be removed).
3. Integrate an additional subject-specific module into the existing scenario: a separate module with subject-specific content (using existing or new customized tasks and outcomes) is created and integrated into a scenario.
4. Add tasks and outcomes: the content of a module can be extended; new customized tasks and outcomes can be created.

Mandatory versus optional: the minimum required documents (outcomes) defined by HERMES to meet governance requirements cannot be removed during tailoring, and the four mandatory modules (Project steering, Project management, Project foundations, Deployment organization) must always be present. Everything else can be selected or deselected.

With these options, additional user-defined, project- or organization-specific scenarios can be modeled. User-defined scenarios can be shared with other HERMES users or made available to all users.

Who and when: tailoring is performed by the project manager (project management) in the initiation phase, as part of selecting and customizing the scenario suitable for the solution development. A combined traditional/agile development approach in particular requires appropriate tailoring in the initiation phase, since the selected scenario and the execution structure plan must be adapted.

### HERMES online, the execution structure plan, and templates

Sizing and tailoring are both supported by HERMES online (the online tool). The project manager selects the scenario in HERMES online in accordance with the decision on next steps. The scenario, with its method components, provides a basic structure that is adopted for solution development and adapted to the specific circumstances of the project. The planning outcomes are recorded in the project management plan, which is created in the initiation phase and continually updated.

Execution planning starts with execution structuring. The execution structure plan is developed with this procedure:

1. Create a study; define scope and boundaries for solution development.
2. In HERMES online: select the scenario and adjust it if necessary (within the scope of the study); create and export the execution structure plan.
3. Integrate the execution structure plan into the project management plan.
4. Supplement solution-specific outcomes and tasks.
5. Adapt roles in the project management plan to the scenario.

In traditional solution development, the execution structure plan is later reviewed and its tasks and outcomes are completed and fleshed out as part of detailed planning before each phase release (rolling planning, from rough to detailed).

## How to Size and Tailor a Project (Checklist)

1. In the initiation phase, conduct the study (initial status report, objectives, rough requirements, solution options) and define the scope and boundaries for solution development.
2. Make the decision on next steps: choose the solution option, choose the approach (traditional or agile), and select the suitable standard scenario based on the study findings.
3. In HERMES online, run sizing: enter the parameters so the tool determines the project value (project size).
4. Let the sizing function produce the size-adapted user-defined scenario, with its execution structure plan and templates.
5. Optionally run tailoring in HERMES online to further customize the scenario, using any combination of the four options: remove modules, remove tasks/outcomes (never the minimum required documents), integrate an additional subject-specific module, add tasks/outcomes. Keep the four mandatory modules.
6. Create and export the execution structure plan; integrate it into the project management plan; supplement solution-specific outcomes and tasks; adapt the roles in the project management plan to the scenario.
7. Download the customized scenario, including the adapted structure plans and templates.
8. Use the project management plan and execution order to compare against the strategies, specifications, and objectives of the core organization, and proceed to the decision on execution release.

## Gaps

- The source range names project sizes generically as "large", "medium", and "small" but does not define discrete size categories with explicit thresholds or labels. The exact set of size categories is not enumerated.
- The "several parameters" that feed the project value are referenced but not individually named or quantified in this source range.
- The detailed inner content of each module (its full task and outcome lists) is described elsewhere in the manual (chapters on Modules, Outcomes, Tasks) and is outside this scenarios/sizing/tailoring scope.
