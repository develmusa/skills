# HERMES Governance and Agile Reference

How HERMES steers projects (governance, decisions, escalation, change control), plans the next phase, runs work in implementation units (traditional) versus iterations and increments (agile), embeds agile development management, reports and manages QA and risk, and relates projects to programs.

## Project governance

Project governance is the part of corporate governance that applies to projects, understood as responsible corporate management and control. It must be implemented above all by the management of the core organization. HERMES is designed to ensure good project governance. The characteristics of good project governance are requirements on project steering and project management:

- Effective and functional project planning, steering, and management.
- Constructive cooperation between the project organization and the core organization.
- Consideration of stakeholder interests, and alignment of objectives with the requirements of the core organization.
- A solid basis for correct, efficient, and transparent decisions, with institutionalized and timely flow of information and transparent communication.
- Comprehensibility of project implementation and of any changes of objectives.
- Appropriate approach to risks.
- Functional and adequate project organization and infrastructure.
- Efficient and sustainable use of resources.

HERMES supports these requirements through several method components: roles, modules and tasks, outcomes (minimum required documents), reporting, phases and milestones, and modules and scenarios. For example, responsibility for tasks and outcomes is assigned to defined roles; the steering and management tasks are grouped in the project steering and project management modules so they are visible to the project sponsor and project management; and minimum required documents such as the execution order and the project management plan provide the documentary basis for steering.

### Decision authority and steering

Within the project, the project sponsor is the highest decision-making authority. The core organization, as owner of the project, fills the decisive roles from its own ranks or the user partner group: first the project sponsor, then (through the project sponsor) the project management and the user representative.

- Project sponsor: defines the framework for project initiation and solution development; determines the assignment of roles (especially the user partner group); decides on the type of development approach (traditional or agile); asserts the interests of the sponsor's core organization; is responsible for continuous communication between the core and project organizations.
- Project management: prepares the steering decisions; safeguards the interests of the core organization; manages the roles of all partner groups; enforces the selected approach; ensures reporting; identifies stakeholders and may gain and inform them.
- User representative: prepares the steering and management decisions; evaluates options in the interest of the core organization; supports the project management in safeguarding core organization interests and in reporting; is responsible for involving stakeholders in project execution.

Two role groups in the core organization act outside the project organization:

- Controlling and compliance bodies: evaluate compliance with the requirements regarding the approach set out in the execution order; review specialist outcomes (for example solution architecture); assess governance.
- Executive board: checks whether the core organization's requirements and objectives for the project are being met; carries overall responsibility for the project portfolio.

Self-determination of users over the project is a governance principle. Because the core organization or user partner group provides the financial resources, it has the right to decide freely (without limitation by other partner groups) on the project structure, the management method, and the approach, and to fill the minimum number of roles with its own or its employed staff. It must actively assert these rights; doing so is part of its obligation. Service providers or developer/operator partner groups must not dominate the project.

### Decision points and quality gates

Decisions are bound to phases, releases, and milestones, which act as quality gates. The decision-making tasks are the formal interface between project organization and core organization. They include: project initiation release, execution release, phase release, release (agile), and project closure, plus decide on solution architecture, decide on ISDP concept, agree on and steer goods/services, and decision on next steps.

- At the end of the initiation phase, the execution order is drawn up; the project sponsor (with the core organization) decides on the execution release, approving continuation and the chosen approach to solution development.
- Before each execution release, release, or phase release, objectives are realigned with the strategies and objectives of the core organization (and any overarching program).
- If at the end of a phase or release the risks are deemed unacceptable, the benefits are lacking, costs are escalating, or feasibility or alignment is missing, a decision on next steps must be made, and the project may be terminated (see decision on project discontinuation).

### Escalation

HERMES does not define a separate escalation procedure beyond the role hierarchy and decision points. Escalation runs along the three hierarchy levels (steering, management, execution) and through the decision-making tasks. The project sponsor is the highest authority within the project; above the project, escalation reaches the core organization (executive board, controlling and compliance bodies) and, where the project is part of a program, the program sponsor. The project sponsor is responsible for resolving conflicts of objectives, and reporting ensures that the competent bodies receive timely information to act.

### Significant changes and change control

In the implementation of corporate planning, significant changes are realized by means of projects. During the initiation phase, more depth is added to the business planning outlines, and the execution order is produced as the basis for the execution release decision. Utilization of the created solution begins with activation (of the product and the organization, of the operation of the system, or of the organization). The project is closed once operation is stable, the solution is accepted, and final tasks are done; the project is then transformed into the application, and in the portfolio only the status changes from project to application (project and application may overlap, especially with agile solutions). The check on whether the original objectives were achieved is carried out during the utilization stage.

Change control during execution is handled by the manage changes task, owned by the project management. Through this task, changes in requirements and scope and their impact on costs, personnel requirements, and deadlines are identified, analyzed, requested, and decided in a timely manner. Changes are recorded in the change status list, and planning (the project management plan) is adjusted accordingly. Comprehensibility of any change of objectives is an explicit governance requirement: decisions are recorded, minutes are taken, lessons learned are recorded continuously, and the project management plan is kept up to date.

## Detailed planning for the next phase (rolling-wave planning)

Planning is the prerequisite for managing and steering the project and is recorded in the project management plan, the key management instrument that holds all the project's plans. It is created in the initiation phase and continually updated. After the decision on next steps, the project management performs execution structuring: it selects the appropriate scenario in HERMES online per the decision on next steps, creates and exports the execution structure plan, integrates it into the project management plan, supplements solution-specific outcomes and tasks, and adapts the roles to the scenario.

After the execution release, the traditional and agile paths diverge. In traditional solution development, planning follows the principle of rolling planning (from rough to detailed, a foundation taken from systems engineering). A master plan is created in the initiation phase and continuously checked and adjusted. Toward the end of the concept and implementation phases, the next phase is planned in detail before the phase release decision, and the rough (master) planning is reviewed. This is done with the prepare phase release task.

Detailed planning for the next phase comprises:

- Review the execution structure plan and complete tasks and outcomes; flesh out tasks and outcomes.
- Define work packages for the next phase and name a person responsible for each; flesh out work package activities and outcomes.
- Verify outlay estimates based on work packages; flesh out resource planning; flesh out the phase schedule.
- Create the decision plan; flesh out the review plan; flesh out the communication plan.
- Update the risk list and measures; verify the overall plan; check the project management plan with a QA measure; coordinate the project management plan with stakeholders.

Work packages are the basis of detailed traditional planning and the prerequisite for controlling and steering. Notes on work packages: several can be created from one task; one or more outcomes result from a work package and are achieved in activities; on completion, the outcomes have passed the QA measures in the review plan or test concept and been accepted; one person is responsible (others may collaborate); a work package typically lasts two to six weeks.

Planning accuracy increases over the course of the project: knowledge about the solution rises and planning inaccuracy falls as outcomes are fleshed out from rough to detailed. HERMES does not prescribe how detailed planning must be at a given point; this is set by the project sponsor and the controlling and compliance bodies, based on the project's situation, characteristics, and complexity. Estimates should state planning accuracy, reserves should be included in the execution order and project management plan, and the estimate assumptions must be documented to meet the transparency requirement.

In agile solution development, the from-rough-to-detailed aspect happens within iterative processing and is carried out autonomously by the execution organization at the execution hierarchy level. Agile release planning is linked to the schedule in the project management plan. On the project side, planning at the management level is limited to coordinating aspects and becomes active again only in the closure phase. If optional release decisions were made mandatory in the initiation phase, the next release is planned in detail at the end of each release with the prepare release closure task.

## Implementation units (traditional approach)

If traditional solution development of an IT project becomes so complex that delivering the whole scope at once seems questionable, or if initial usable outcomes should be delivered as quickly as possible, the implementation and deployment phases can be broken down into several implementation units. HERMES allows implementation units to run sequentially, overlapping in time, or in parallel.

- The initiation and concept phases are completed first. Implementation units can start after the concept phase; from that point the project runs in the phases and milestones of the respective implementation unit, with no superordinate phase model.
- Each implementation unit covers the implementation and deployment phases, comprises all technical and organizational outcomes required to deploy the system or a part of it, and ends with the product or system being used productively.
- The release of the first implementation unit is the implementation phase release, which requires regular closure of the concept phase. The start of each implementation unit must be released by project steering and requires an updated project management plan. Each implementation unit goes through the decision-making tasks of steering and management.
- The number of implementation units is not limited by HERMES, but the project duration should not be unlimited; implementation units are therefore planned as a whole in the concept phase.
- From a controlling perspective, implementation units are planned and controlled separately for costs, deadlines, and outcomes. They form independent control units, and reporting should be geared to them. A final evaluation of each implementation unit should be carried out and its lessons learned documented and used.
- At the end of the last implementation unit, a decision is made on effective release of the closure phase. The closure phase carries out its tasks and outcomes, including the final project evaluation of all implementation units.

## Agile development management

Agile methods (Scrum, Kanban, and scaled methods such as SAFe up to the Essential or ART Flow level) are development management methods, not project management methods. On their own they do not fulfill HERMES framework conditions (compliance with governance, smooth embedding in the core organization's planning and controlling processes, common language and uniform terminology). HERMES therefore embeds agile methods inside its project management framework. The agile method is inserted like a black box: HERMES does not detail the encapsulated development approach but defines the interfaces for steering, management, communication, and reporting (the relevant outcomes and specific roles).

Under the agile approach the project runs in three phases: Initiation, Execution, Closure. The understanding of roles draws on both traditional project management and agile development methods:

- The project organization works according to traditional project management; the execution organization works according to the chosen agile method and organizes its own work.
- The traditional and agile approaches differ fundamentally in how the execution hierarchy level is managed. In the traditional approach the project manager issues work orders. In the agile approach the work of the execution organization is steered by the user representative via the solution requirements, and the teams self-organize. The project manager manages the project but must not interfere with the self-organization of the agile execution organization.
- The user representative acts as the interface between the project organization and the execution organization, and is the contact person for the project manager. Internal agile terminology (for example product owner, backlog, sprint) is not prescribed by HERMES; it depends on the development method used. Only the outcome interfaces and the project management terminology are fixed.

Agile development management determines the organization and steering of the execution organization and autonomously steers solution development within the predefined framework. The method-specific role models, processes, and rituals can be practiced without interference, provided there is consensus within the core and project organizations. Further detail is in the extension document "Agile Guide - Project management" ("Agiler Leitfaden - Projektmanagement").

### Releases, iterations, and increments

In the agile approach, work is organized into releases. The optional decision on release creates the prerequisite for work in the next release; whether it is mandatory for a given project is set by a note in the project management plan (decided by the project sponsor in the initiation phase). At the end of the current release, the release report is accepted and a decision is made to close the release; the sponsor then decides on the next release. Before release, the release report and project management plan are coordinated with the overarching strategies and objectives of the core organization or any overarching program, and new findings are taken into account. If the set goals cannot be achieved, the project is terminated.

### Combining traditional and agile

The choice of approach (traditional or agile) is made at the project level because each project has its own characteristics; this also applies to projects within a program, and even parallel IT projects may need different approaches. The approach cannot be dictated by the program or portfolio, and the choice must be comprehensible and meet the requirements of controlling and compliance bodies. In practice a project may combine both approaches. This requires appropriate tailoring in the initiation phase, since the selected scenario and the execution structure plan must be adapted.

## Reporting, QA, and risk governance

Reporting fulfills the transparency and traceability governance requirements. It is carried out periodically along the phases according to the core organization's requirements and formally regulates the flow of information within the project organization and toward the core organization. Reporting stays uniform (encapsulated) toward the core organization regardless of the development approach, so governance in reporting is maintained for both traditional and agile development.

Report types:

- Project status report: prepared periodically from the start of the project until closure; the project manager informs the project sponsor and core organization about project status (plan/actual comparison) and the expected next steps (forecast). It contains the current risk assessment.
- Phase report: at the end of the concept, implementation, deployment, and execution phases, the phase outcomes and the planning of the further course are prepared so the project sponsor can decide on how to proceed (usually phase release).
- Release report (agile): at the end of each release during the execution phase, the release outcomes are prepared so the sponsor is informed about the success of the release and the overall development; if release decisions were made mandatory in the initiation phase, it serves as the decision-making tool.
- Final project evaluation: created at the end of the closure phase; provides the basis for continual improvement based on lessons learned (plan/actual comparison, key findings recorded).

Supplementing reporting, defined specialist outcomes (for example solution architecture) are forwarded to the controlling and compliance bodies for review.

QA and risk governance:

- Risk management is continuously managed with the manage risks task; the project status report carries the current risk assessment and the project management's view.
- At the project steering hierarchy level, the quality and risk manager role supports the project sponsor with an independent assessment of the project. In the agile environment, where the budget is fixed, the sponsor may appoint an independent quality and risk manager for steering support because financial control and success are measured by other instruments (for example a burndown chart comparing estimated and actual remaining outlay).
- The QA plan and review plan are created in the project management plan; QA measures are applied to outcomes (review plan or test concept) and to the project management plan itself.
- At the end of a phase or release, unacceptable risks can trigger a decision on next steps and possibly project termination.

## Program management and projects

Program management is an extension of project management for organizations with far-reaching, comprehensive changes that must achieve objectives across a group of interrelated projects in a lean, coordinated way. In a program, projects are grouped together. Projects and programs can be managed side by side in a core organization, and within a portfolio a project can be stand-alone or part of a program (a program contains several projects). HERMES gives project and program management a common understanding and terminology; the full treatment is in a separate program management appendix.

Relationship from the project perspective:

- The program sponsor steers the program. The program management manages the program and coordinates the cross-project aspects and dependencies between projects. Each project is managed by its own project management, and the user representative defines the solution.
- Steering can be supported by a project committee (under the project sponsor) and, at a superordinate level, by a program committee (under the program sponsor). From the controlling and compliance perspective, each project remains an independent controlling object with its own specifications for costs, time, and outcomes.
- The program closure program phase can be released only once all projects are completed.
- If a project becomes part of a program, the project organization must be integrated into the program organization and some roles (mainly project sponsor and project management) are adjusted or replaced. The effects are mainly in steering, management, and control. The adjusted role descriptions for the program are recorded in the project management plan.
- If a project is embedded in a program, the program management plan takes precedence; project provisions of the project management plan must not run counter to the program provisions. The one exception is the choice between the traditional and agile approach, which must always be examined per project and documented in the study.

Two program organization forms are described from the project view:

- Program organization 1: several core organizations are responsible for their assigned projects. The project sponsor remains responsible for the project's success and steers it on behalf of the program sponsor, but each project sponsor reports both to its own core organization and to the program sponsor and must take both into account. The program management runs and coordinates the project and project management from the overarching program perspective and continuously coordinates with the project sponsor; the project management executes the program-specific instructions and coordinates the project management plan with the program management.
- Program organization 2: one core organization is responsible for the program and all its projects. There is no project-level sponsor; the program management replaces the project sponsor and assumes all the sponsor's tasks and duties within the project. The program management sits at the steering hierarchy level in the project organization and at the management level in the program organization; the project management is subordinate to it. Responsibility for project success lies with the program sponsor.
